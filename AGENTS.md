# AGENTS.md

## Package manager

Use **pnpm**. Lockfile is `pnpm-lock.yaml`. `engine-strict=true` in `.npmrc`.

## Key commands

```bash
pnpm dev          # SvelteKit dev server (demo/test harness only — not the library build)
pnpm check        # svelte-kit sync + svelte-check (type-check everything)
pnpm lint         # prettier --check + eslint
pnpm format       # prettier --write
pnpm test         # vitest --run (single pass, both projects)
pnpm test:unit    # vitest (watch mode)
pnpm prepack      # builds the published library: svelte-kit sync → svelte-package → publint
pnpm build        # vite build + prepack (rarely needed; prepack is the important step)
```

`prepack` is the library packaging step. Run it to verify the `dist/` output is valid before publishing.

## Project structure

SvelteKit is only a **dev/preview harness**. The published artifact is built by `svelte-package` from `src/lib/`.

- `src/lib/index.ts` — library entry point (currently exports nothing; add exports here)
- `src/lib/components/` — empty; this is where components go
- `src/lib/*.svelte.ts` — Svelte module files (runes-capable TS, see below)
- `src/routes/` — demo/dev app only, not published
- `dist/` — generated by `prepack`, not committed

## Svelte 5 runes — forced on

`svelte.config.js` sets `runes: true` for every file outside `node_modules`. There is no opt-out. Write all components and `.svelte.ts` modules using runes syntax.

## `.svelte.ts` extension

To use Svelte 5 runes (`$state`, `$derived`, `$effect`, etc.) in a plain TypeScript file (not a `.svelte` component), the file **must** use the `.svelte.ts` extension. Example: `table.svelte.ts`. A plain `.ts` file will not have runes compiled.

## Testing

Two Vitest projects are defined inside `vite.config.ts` (no separate `vitest.config.ts`):

| Project  | Environment             | File pattern                         |
| -------- | ----------------------- | ------------------------------------ |
| `client` | Chromium via Playwright | `*.svelte.{test,spec}.{js,ts}`       |
| `server` | Node                    | `*.{test,spec}.{js,ts}` (non-svelte) |

The **double extension** (`.svelte.spec.ts`) is what routes a test to the browser project. Use `vitest-browser-svelte` to render components in browser tests.

**`requireAssertions: true` is enforced globally.** Every `it()` / `test()` block must contain at least one `expect()` call or Vitest will fail it.

## Formatting conventions

Prettier config (`.prettierrc`):

- **Tabs** (not spaces)
- **Single quotes**
- **No trailing commas**
- Print width: 100

## Tailwind CSS v4

No `tailwind.config.js`. Tailwind is configured entirely via CSS. The entry stylesheet is `src/routes/layout.css` which contains only `@import 'tailwindcss'`. Prettier's `tailwindStylesheet` is set to this file for class sorting.

## Bleeding-edge tooling

This project tracks very recent releases: TypeScript 6, Vite 8, Vitest 4. If a type error or API looks wrong, check whether the upstream package has changed its API rather than assuming the code is wrong.

## Project status

Early stage. `src/lib/components/` is empty and `src/lib/index.ts` exports nothing. Core state logic lives in `src/lib/table.svelte.ts`.
